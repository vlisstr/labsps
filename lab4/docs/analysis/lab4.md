# Лабораторна робота №4 — аналіз

## 1. Що змінилося порівняно з lab 3

У lab 3 Application Layer був замкнений: handler виконував основну
операцію і повертав керування. Все, що ставалося «після» — нікого
не цікавило.

У lab 4 ми додали **побічну операцію** (audit) і два способи її викликати:

| Що з'явилося | Де живе | Призначення |
|---|---|---|
| `src/audit/` — окремий модуль | `AuditEntry`, `InMemoryAuditLogger`, subscribers | Допоміжний компонент (записує журнал) |
| `IAuditLogger` (порт) | `application/ports/` | Контракт для прямого виклику audit-у в sync-режимі |
| `IEventBus` (порт) | `application/ports/` | Контракт для публікації подій |
| `InProcessEventBus` (адаптер) | `infrastructure/event_bus/` | In-process pub/sub реалізація |
| Integration Events | `application/events/` | `UserRegistered`, `ItemCreated`, `ItemUpdated`, `ItemDeleted` |
| `mode` у Command Handler | `application/commands/*/*Handler.js` | Перемикає sync ↔ async без зміни контролерів |
| `GET /audit` | `presentation/routes/auditRoutes.js` | Демонструє результат побічних ефектів |

Базова бізнес-логіка не змінилася: домен з lab 2 і CQS з lab 3 — на місці.

## 2. Які побічні операції виділив і чому

Обрано **Audit Log** — журнал «хто, коли, що зробив»:
- `user.registered` (при `POST /auth/register`)
- `item.created` (при `POST /items`)
- `item.updated` (при `PUT /items/:id`, разом з `changes`)
- `item.deleted` (при `DELETE /items/:id`)

**Чому саме audit:**
1. **Класичний side-effect.** Аудит не потрібен клієнту в response; його
   читає security/support окремо.
2. **Без зовнішніх інтеграцій.** Можна реалізувати чесно in-process,
   без mock-у поштовиків чи аналітики.
3. **Дозволяє продемонструвати ідемпотентність.** Подія приходить
   двічі — журнал не повинен мати дублів. Це той випадок, де
   ідемпотентність насправді бачиться.
4. **Логічно окремий модуль.** `src/audit/` не залежить від `domain/`
   і `application/commands/`, тільки від портів — повна ізоляція.

## 3. Порівняння sync і async

### 3.1. Час відповіді API

У sync-режимі: `T(POST /items) = T(create item) + T(audit.record)`.
У моєму in-memory сетапі audit додає мікросекунди — непомітно.
Але якщо audit пише в окрему БД через мережу (типовий випадок), додаються
сотні мс на кожний коміт.

В async-режимі: `T(POST /items) = T(create item) + T(eventBus.publish)`,
де `publish` лише викликає підписників у тому ж процесі. У моєму
тесті — стабільні десятки мікросекунд. Якщо підписник важкий (наприклад,
відправляє email через SMTP), клієнт цього вже не відчуває.

### 3.2. Поведінка при збоях

| Сценарій | Sync | Async |
|---|---|---|
| Audit БД недоступна, нова реєстрація | Handler ловить помилку у try/catch, логує і повертає 201. Аудит втрачено. | Подія опублікована; підписник падає; основна операція — 201. Подія потенційно ретранслюється outbox-ом (у нас поки що — втрачена після крашу процесу). |
| Підписник кидає виняток | — | Event bus ловить, інші підписники продовжують працювати (verified тестом). |
| Множинна доставка | — | `InMemoryAuditLogger` дедуплікує за ключем (verified тестом). |

Тест `CreateItemSyncVsAsync.test.js` явно перевіряє це поведінкове
розмежування — обидва режими ізолюють основну операцію від збоїв audit,
але способами, різними з точки зору операційної моделі.

### 3.3. Зв'язаність

| Що знає Handler | Sync | Async |
|---|---|---|
| Про існування Audit | Так (тримає `IAuditLogger`) | Ні |
| Про структуру `AuditEntry` | Так (заповнює `action`, `entityType`, …) | Ні |
| Про список підписників | — | Ні (Event Bus не розкриває) |
| Додавання нової побічної операції (аналітика) | Нова залежність у конструкторі, ще один виклик у методі | Нуль змін у handler-і. Додаємо `bus.subscribe('ItemCreated', analytics)` у composition root |

Це і є головна вигода async-варіанта: відправник не знає одержувачів.

### 3.4. Складність реалізації та тестування

**Sync легше:**
- Один call stack, можна `console.trace` і одразу видно.
- Тест: `assert.equal(auditLogger.size(), 1)` після виклику handler.
- Не треба думати про ідемпотентність — кожен виклик унікальний.

**Async складніше:**
- Треба тестувати окремо: handler публікує подію (через `CapturingEventBus`);
  підписник правильно обробляє подію (через прямий `eventBus.publish`);
  end-to-end (handler + bus + subscribers).
- Треба думати про повторну доставку — отже потрібен `event_id` або ключ
  ідемпотентності.
- Якщо підписник асинхронний (наприклад, посилає HTTP) — треба чекати або
  робити інтеграційний тест.

У цій лабі async реалізовано in-process, тому асинхронність відносна:
підписники виконуються одразу під час `await publish()`. Це достатньо,
щоб показати pub/sub-зв'язаність. У реальній системі (RabbitMQ/Kafka)
додаються мережа, ретраї і dead-letter черги.

## 4. Який підхід для production

Для **моноліту з 1 БД** (як цей проєкт) обираю **async через in-process Event Bus**:
- Дешева ціна впровадження (`InProcessEventBus` — це 30 рядків коду).
- Handler-и тонкі, додавання нових підписників не їх зачіпає.
- Збій підписника не ламає основну операцію.

Для **розподіленої системи** (audit/notification — окремі сервіси) обираю
**async через брокер + Outbox**:
- `InProcessEventBus` замінюється на `RabbitMqEventBus` (інтерфейс той самий).
- Між `INSERT` у БД і `publish` має бути транзакційний Outbox, інакше
  падіння процесу між цими діями втратить подію.
- Підписники мають дедуплікувати за `event_id`, бо брокер дає at-least-once.

**Sync** залишаю для операцій, де потрібна **strong consistency**:
- Списання коштів перед підтвердженням замовлення.
- Перевірка ліцензії перед видачею токена.
- Будь-що, де клієнт має побачити кінцевий результат у тій же відповіді.

Тобто в реальному коді обидва підходи співіснують. У цій лабі ми
використали один (audit) як приклад, але архітектура готова до
будь-якого розкладу.

## 5. Що станеться при подвійній доставці

В async-режимі повторна доставка можлива:
- Брокер передоставив подію після timeout-а підтвердження;
- Outbox-processor перечитав ту саму подію після рестарту;
- Підписник встиг записати у свою БД, але крашнувся до ACK.

Я зробив підписники ідемпотентними двома способами:

1. **Ключ ідемпотентності в `AuditLogger`.** `InMemoryAuditLogger`
   тримає `Set` з ключем `(action, entityType, entityId, occurredAt)`.
   Якщо подія приходить вдруге з тим самим часом — другий запис не
   створюється. Тест `AuditSubscribers.test.js → «дубль ігнорується»`
   це доводить (3 публікації → 1 запис у журналі).

2. **`occurredAt` генерується один раз у Command Handler.** Це робить
   подію дійсно повторюваною: при retry той самий `occurredAt` потрапляє
   у ключ дедуплікації. Якби ми генерували час всередині підписника,
   ідемпотентність би не працювала.

Альтернативи на майбутнє:
- Дати кожній події `eventId: uuid()` і дедуплікувати за `eventId`.
  Це необхідно, якщо одна й та сама операція може законно повторюватися
  (наприклад, повторні оновлення).
- «Природна» ідемпотентність — `UPSERT`-операції замість `INSERT`/`UPDATE`.
  Не підходить для журналу (append-only).

## 6. Висновки

Lab 4 додав до проєкту:
- Окремий модуль `audit/` як приклад «допоміжного компонента»;
- Інфраструктуру для подій (`InProcessEventBus`) і чотири
  Integration Events;
- Дві гілки в Command Handler-ах: sync і async, перемикання через
  одну змінну середовища.

Найважливіший практичний урок: **спосіб комунікації — це trade-off на
рівні операції, не системи**. Sync для оплати, async для нотифікацій —
це нормально в одному додатку. Архітектура lab 4 готова до такого
гібриду, бо одиниця рішення — це конкретний Command Handler з його
налаштуванням `mode`, а не глобальний прапор.

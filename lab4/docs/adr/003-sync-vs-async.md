# ADR-003: Asynchronous communication via in-process Event Bus

- Статус: Прийнято
- Дата: 2026-05
- Контекст: Лабораторна робота №4 — додати побічну операцію (audit log)
  і реалізувати обидва режими комунікації.

## Контекст

У lab 3 Command Handler виконував лише основну операцію (зберегти, видалити,
оновити). Тепер бізнес каже: «треба фіксувати у журналі, хто і коли що зробив».
Це класична побічна операція — її результат не потрібен клієнту відразу і її
збій не повинен зривати бізнес-операцію.

Є два варіанти підключення:

### A) Sync — Command Handler напряму викликає auditLogger
```js
await items.save(item);
await auditLogger.record(new AuditEntry({...}));
return item.id;
```
**Плюси**: просто, легко відлагоджувати, видно весь call stack.
**Мінуси**: handler знає про audit; додаємо аналітику — лізе ще одна
залежність у конструктор; час відповіді = main + audit; збій audit
вимагає рішення на місці (re-raise / log / silent ignore).

### B) Async — Command Handler публікує подію, audit підписується
```js
await items.save(item);
await eventBus.publish(new ItemCreated({...}));
return item.id;
```
**Плюси**: handler знає лише про `IEventBus`; додавання підписників
(analytics, notifications, sync з зовнішньою системою) — нуль змін у
handler; збій підписника не зриває основну операцію; події можна
переадресувати на брокер коли знадобиться.
**Мінуси**: eventual consistency, ідемпотентність обробників, складніше
відлагоджувати.

## Рішення

Обираємо **B: async через in-process Event Bus** як режим за замовчуванням,
але код підтримує і `sync` (через змінну `COMMUNICATION_MODE`) — щоб
порівнювати поведінку і використовувати sync як «аварійний» режим,
якщо подієва доставка зламається.

In-process bus достатньо, бо аудит — це той самий процес, гарантій
персистенції повідомлень нам поки не треба. Якщо колись audit винесеться
у окремий сервіс, замінимо `InProcessEventBus` на RabbitMQ/Kafka-адаптер
(і додамо Outbox для надійної доставки) — інтерфейс `IEventBus` не зміниться.

## Класифікація: чи це «доменна подія»?

Ні. `ItemCreated`, `UserRegistered` тощо тут — **Integration Events**:
вони описують факт, який вже стався, і призначені для інших модулів
(audit, потенційно analytics, notifications). Доменні події живуть
всередині агрегату; ми ж публікуємо їх з Application Layer, ПІСЛЯ
успішного збереження. Семантично це integration events.

## Ідемпотентність

Підписник `InMemoryAuditLogger` дедуплікує записи за ключем
`(action, entityType, entityId, occurredAt)`. Це гарантує, що
повторна доставка тієї самої події не створить дубль запису.

Стратегія підходить для in-process bus, де `occurredAt` генерується
одного разу handler-ом і входить у саму подію. Для зовнішнього брокера
варто додати окремий `event_id` (UUID) і дедуплікувати за ним.

## Наслідки

- З'явилися два нових порти: `IEventBus`, `IAuditLogger`.
- З'явився новий модуль `src/audit/` з власним контрактом і набором підписників.
- Command handler-и мають дві гілки виконання (`sync`/`async`), увімкнення
  через `mode`-параметр, що передає `container.js`.
- Інтерфейс `IEventBus` дозволяє в майбутньому замінити транспорт на брокер
  без зміни handler-ів.
- Аудит видимий через `GET /audit` — корисно для перевірки роботи async-режиму.

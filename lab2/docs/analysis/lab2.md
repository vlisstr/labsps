# Лабораторна робота №2 — аналіз

## 1. Що змінилося порівняно з lab 1

**Lab 1 (вихідний стан).** Чотири файли (`server.js`, `routes/auth.js`,
`routes/items.js`, `middleware/auth.js`) і два глобальні масиви `users[]`,
`items[]`. Кожен обробник маршруту самостійно:

- читає `req.body`;
- валідує наявність полів;
- хешує пароль bcrypt-ом;
- перевіряє існування користувача / власника;
- кидає HTTP-помилки (`res.status(401).json(...)`).

Логіка, валідація і I/O змішані; будь-яка зміна правила («дозволити змінювати
item ще й адміну») зачіпає одночасно маршрут, валідацію і JWT-перевірку.

**Lab 2 (після рефакторингу).** Чотири шари з суворим правилом залежностей:

```
Presentation → Application → Domain ← Infrastructure
```

| Раніше було                                  | Тепер живе у                                       |
|----------------------------------------------|----------------------------------------------------|
| перевірка формату username у роуті           | `Username` (VO, `src/domain/user/Username.js`)     |
| `users.find(u => u.username === ...)`        | `IUserRepository.findByUsername()` + Sqlite/Memory |
| ручне порівняння `item.ownerId !== sub`      | `Item.ensureCanBeDeletedBy()` + `ForbiddenError`   |
| `bcrypt.hash` напряму в роуті                | `IPasswordHasher` (порт) + `BcryptPasswordHasher`  |
| `res.status(409).json({ error: 'exists' })`  | `ConflictError` → `errorHandler` мапить у 409      |
| `users.find(...).id` як int із масиву        | `nextId()` через репозиторій                       |
| `passwordHash` як `string` у структурі       | `PasswordHash` VO з `toJSON() = '[PasswordHash]'`  |

З'явилися артефакти, яких у lab 1 не було взагалі: `UserFactory`, `ItemFactory`,
Value Objects, маппери Domain ↔ Record, ADR, doменні помилки, інтерфейси
репозиторіїв і композиційний корінь (`container.js`).

## 2. Переваги розділення на шари

- **Інваріанти не можна обійти.** У lab 1 `item.ownerId !== req.user.sub`
  треба було пам'ятати написати у двох місцях (PUT і DELETE). У lab 2
  будь-яка модифікація Item проходить через `_assertOwner(actorId)`
  всередині агрегату — забути неможливо.
- **Безпека за замовчуванням.** `PasswordHash.toJSON()` повертає
  `'[PasswordHash]'`, тому навіть випадковий `res.json(user)` не зливе хеш.
  У lab 1 така помилка була в одному `console.log` від продакшну.
- **Тести бізнес-логіки без БД і HTTP.** Unit-тести домену (`Username`,
  `Item`, `UserFactory`, use cases) запускаються через `node:test` за
  ~100 мс, без `npm install` зовнішніх раннерів — лише вбудований модуль.
  Інтеграційні тести підіймають Express на ефемерному порту й бігають
  окремою командою.
- **Заміна БД — точково.** Перемикання `memory ↔ sqlite` — це одна змінна
  оточення `PERSISTENCE`. Жодний файл у `domain/` або `application/`
  при цьому не змінюється.
- **Помилки мають семантику.** `errorHandler` — єдине місце мапінгу:
  `ValidationError → 422`, `ConflictError → 409`, `ForbiddenError → 403`,
  `AuthenticationError → 401`, `NotFoundError → 404`. Додати gRPC-шар можна
  без перетягування `HTTPException` через домен.

## 3. Недоліки і ускладнення

- **Більше файлів і коду.** Замість двох роутів і однієї мідлвари стало
  ~25 модулів. Для CRUD на дві сутності це об'єктивно багато; виправдано
  лише наявністю інваріантів (власник, формат, унікальність).
- **Маппінг між шарами.** Доводиться писати `UserMapper.toDomain/toRecord`,
  `userToView`, плюс окремо `UserRecord` як «ORM-сирий рядок». Це додатковий
  код, який нічого не «робить», крім перетворення форм даних. Окупається,
  коли поле в БД (snake_case, NULL) починає розходитись з доменним.
- **Складніша навігація.** Щоб зрозуміти, що відбувається при PUT `/items/:id`,
  треба пройти: маршрут → use case → `Item.rename` → репозиторій → маппер.
  Для нових учасників проєкту потрібен ADR і ця сторінка.
- **Дисципліна імпортів.** Один випадковий `require('bcryptjs')` у домені
  ламає всю архітектуру. Захист — короткий grep у README і ревʼю.

## 4. Наскільки простіше змінити БД або фреймворк

**Заміна БД (in-memory → SQLite → Postgres).**
- `domain/` і `application/` — нуль змін.
- Треба написати лише `PostgresUserRepository` / `PostgresItemRepository`
  (≈ 50 рядків кожен) і додати гілку в `container.js`.
- Тести use cases і доменні не вимагають перезапуску — вони працюють з
  фейковими репозиторіями.

**Заміна фреймворку (Express → Fastify, або додавання gRPC).**
- `domain/`, `application/` і `infrastructure/` — нуль змін.
- Переписується лише `presentation/`: новий `buildApp`, новий
  `errorHandler` (Fastify має свій). Use cases викликаються тими самими.
- gRPC-шар можна додати **паралельно** з HTTP, бо обидва — це лише
  тонкі обгортки над одним і тим же набором use cases.

В lab 1 і те і те потребувало б одночасної правки усіх роутів.

## 5. Чому обрано Rich Domain Model

Деталі — в `docs/adr/001-rich-domain-model.md`. Стисло:

- Інваріант «власник може змінювати» — точка, де Anemic-модель
  систематично «забуває» правило в одному з шляхів коду.
- `PasswordHash` як VO усуває цілий клас помилок (витік хешу в логах/JSON).
- Доменні тести стають дешевими: один `node:test` без БД, без bcrypt,
  без jwt — і ми вже впевнені у бізнес-правилах.
- Anemic-модель була б виправдана, якби це був чистий «таблиці-в-JSON»
  CRUD без бізнес-правил. Тут це не так.

## 6. Висновки

Рефакторинг показав, що навіть на CRUD-двушці DDD-патерни окупаються в
трьох конкретних точках: захист `passwordHash`, інваріант власника Item-а
і дешеве unit-тестування. Ціна — приблизно п'ятикратний приріст файлів
і явний маппінг між шарами; ця ціна стає економічно виправданою саме
завдяки повній ізоляції доменного шару від інфраструктури.

# ADR-001: Обрати Rich Domain Model замість Anemic

- Статус: Прийнято
- Дата: 2026-05
- Контекст: Лабораторна робота №2 — рефакторинг lab1 у шарову архітектуру з DDD.

## Контекст

Lab 1 — це CRUD з двома сутностями (`User`, `Item`) і JWT-логіном.
Інваріантів небагато, але вони не тривіальні:

1. **Username**: довжина 3–32, обмежений набір символів, нормалізація до lowercase.
2. **Пароль**: ніколи не зберігається в plain-текст; plain-пароль має обмеження
   мінімальної довжини; хеш не повинен витікати у логи/JSON.
3. **Унікальність username** — крос-агрегатний інваріант, потребує сховища.
4. **Власник Item-а**: тільки автор створення може перейменувати, оновити чи видалити.
5. **Формати ItemName / ItemDescription**.

Якщо обрати **Anemic Model**, ці правила розповзуться по `ItemService`,
`UserService`, контролерах і валідаторах. Тоді легко з'являться шляхи,
де хтось «забув» викликати перевірку — і інваріант буде порушено.

## Розглянуті варіанти

### A) Anemic Domain Model
- `User` / `Item` — публічні структури з полями.
- Всю логіку (валідація, перевірка власника, хешування) тримає `ItemService`,
  `UserService`, валідатори запитів.

**Плюси**: менше класів, простіше «увійти» новачку, схоже на типовий MVC-Express.
**Мінуси**:
- Логіку «власник може лише сам редагувати» доводиться повторювати у `update`
  і `delete` — легко забути.
- `passwordHash` як `string` дозволяє випадково віддати його у JSON-відповіді.
- Інваріанти VO (формат username) живуть або у DTO-валідаторі, або в моделі БД —
  обидва місця слабкі (DTO не запускається при імпорті з БД).
- Тести бізнес-логіки змушені піднімати «сервіси» з купою mock-залежностей.

### B) Rich Domain Model (обрано)
- `Username`, `PasswordHash`, `ItemName`, `ItemDescription` — Value Objects, що
  самі захищають свою валідність.
- `User`, `Item` — entity з приватними полями і методами `rename`, `changePassword`,
  `ensureCanBeDeletedBy`. Не існує способу створити невалідний об'єкт.
- Складні інваріанти (унікальність username) — у `UserFactory`, який отримує
  репозиторій через DIP.
- Application Layer оркеструє: завантажити → викликати метод моделі → зберегти.

**Плюси**:
- Інваріанти централізовані у моделях; помилку власника / валідації НЕМОЖЛИВО
  обійти випадково (метод просто кидає `ForbiddenError` / `ValidationError`).
- `PasswordHash` як VO з `toJSON()` гарантує, що хеш ніколи не з'явиться у відповіді.
- Доменні тести — pure unit, без БД, фреймворку чи зовнішніх бібліотек.
- Природне місце для майбутніх правил (наприклад, ліміт N items на користувача —
  один новий метод у factory).

**Мінуси**:
- Більше класів (VO, Entity, Factory) — вища стартова вартість.
- Постійне «обгортання примітивів у VO» вимагає дисципліни.
- Маппінг доменна модель ↔ ORM-запис — додатковий код у Infrastructure Layer.

## Рішення

Обираємо **Rich Domain Model**. Інваріанти (особливо власник Item і захист
`passwordHash`) — це саме той випадок, де додаткові класи окупаються одразу
й суттєво підвищують безпеку API.

## Наслідки

- У домені з'являються VO (`Username`, `PasswordHash`, `RawPassword`,
  `ItemName`, `ItemDescription`).
- Сутності (`User`, `Item`) інкапсулюють поля, доступ через геттери;
  всі зміни проходять через доменні методи.
- `UserFactory` залежить від `IUserRepository` та `IPasswordHasher` —
  саме завдяки цьому складний інваріант унікальності залишається у домені.
- Application-сервіси (use cases) перетворюються на тонкі оркестратори.
- Для тестів пишуться фейкові реалізації портів — і доменні тести бігають
  миттєво, без БД та HTTP.

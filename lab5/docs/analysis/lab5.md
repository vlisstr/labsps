# Лабораторна робота №5 — аналіз та ретроспектива курсу

## Частина 1. Аналіз лабораторної 5

### 1.1. Як визначив межі контекстів

Я почав з принципу «**один контекст = одна причина для змін**».
В системі є три такі причини:

1. **Бізнес-правила** (хто є user, що таке item, хто може його змінити) →
   Core.
2. **Прозорість і ретроспектива** (хто/коли/що зробив у системі) → Audit.
3. **Звітність і агрегація** (топ users, активність за добу) → Analytics.

Кожна з цих причин має різного «замовника»: Core — це продукт, Audit — це
security/compliance, Analytics — це менеджмент/маркетинг. Зміни в одному
не повинні чіпати інших. Це підтвердження, що межі обрані правильно.

### Альтернативи, які я розглядав

- **Залишити одним модулем** — мінімалізм, але не задовольняє лаба. Крім того,
  audit з lab 4 вже структурно жив окремо, а analytics — це чистий новий
  bounded context з власною моделлю.
- **Виділити Auth в окремий контекст** — теоретично правильно (DDD: auth = окремий
  context), але технічно тісно пов'язано з User aggregate. Зробив би
  spanning-агрегат, що зашкодило б навчальній ясності.
- **Зробити сторінку «items per resource type»** — додатковий read-model
  всередині Analytics. Вирішив не ускладнювати: вже наявні `UserStats`,
  `DailyMetric` достатньо демонструють патерн.

### 1.2. Як ACL захищає модулі від змін

ACL у мене — це `auditEventHandlers.js` у Audit і `CoreEventTranslator.js` у Analytics.
Вони перекладають **зовнішню форму** (структура Core-події) у
**внутрішню модель** модуля-споживача.

**Сценарій захисту: Core перейменує `itemId` → `id` у `ItemCreated`.**

- Без ACL: всі підписники зламалися б, бо звертаються до `event.itemId` напряму.
- З ACL: ламається лише ACL. У Audit/Analytics всі підписники працюють далі
  через `userStats.recordItemCreated(occurredAt)` — їм байдуже, як в Core
  називається поле з ID.

ACL — це **точка лежить-всередині-споживача** (per lab requirement
«ACL належить споживачу»). Це важливо: якщо Core хоче змінити свій
контракт, Core нічого не повинен робити з ACL — це Audit/Analytics
доведеться оновити переклад. Ця відповідальність правильно лежить на
тому, хто залежить.

### 1.3. Де strong, де eventual і чому

| Точка | Тип | Обґрунтування |
|---|---|---|
| `Item.rename(actorId)` + `items.save(item)` | **Strong** | Одна транзакція в межах агрегату. Інваріант «власник може редагувати» перевіряється до save. Поза цією точкою інваріант не існує — отже він має бути strong-захищеним. |
| Core → Audit (`UserRegistered` → запис у журнал) | **Eventual** | Audit оновлюється async. У момент між `core.publish()` і `audit.handle()` система «знає» про реєстрацію в Core, але не в Audit. Це нормально: журнал — пост-фактум по визначенню. |
| Core → Analytics (`ItemCreated` → інкремент лічильника) | **Eventual** | Те саме. Якщо `GET /analytics/summary` повернув на 50 мс «застарілу» цифру — це некритично. |
| Між підписниками всередині однієї події | **Не визначено** | Audit і Analytics обидва підписуються на `ItemCreated`. Один з них може закінчити обробку раніше іншого. Це не проблема, бо їхні дані не корелюються між собою. |

**Чому eventual прийнятна в цих випадках:**
- Бізнес-операція успіх не залежить від audit/analytics — клієнт отримує
  201 одразу після Core-save, незалежно від побічних модулів.
- Aудит та аналітика не є read-after-write точками для клієнта.
  Він не пише item і відразу не запитує статистику з очікуванням
  негайної відповіді.
- Якщо Audit крашнеться — Core і Analytics продовжують працювати
  (verified: `ModulesIntegration.test.js → «Збій audit-підписника не зриває Analytics»`).

### 1.4. Що зміниться, якщо винести модуль у окремий сервіс

Найбільший кандидат — **Analytics** (типовий шлях: «зробимо аналітичний
data warehouse»).

**Що зміниться:**
1. **Транспорт події** — `InProcessEventBus` заміниться на
   `RabbitMQEventBus` (або Kafka). Інтерфейс `IEventBus` лишається тим
   самим — це адаптерна заміна, без зміни жодного бізнес-коду.
2. **Outbox у Core** — між `repo.save()` і `bus.publish()` зараз вікно
   втрати події при краші. Для внутрішнього bus це прийнятно. Для
   зовнішнього — обов'язковий transactional outbox, інакше Core закомітить
   зміну, а Analytics її ніколи не побачить.
3. **ACL посилюється** — зараз `CoreEventTranslator` отримує JS-об'єкт.
   З брокером він отримуватиме байти, які треба десеріалізувати. Десериалізатор
   стає частиною ACL.
4. **Authentication/authorization для analytics** — зараз модулі живуть в
   одному процесі і мають спільний контекст. У окремому сервісі знадобиться
   власна автентифікація (service-to-service: API key, mTLS чи JWT з shared
   secret).
5. **HTTP routes Analytics виносяться в окремий процес.** Шар `composition_root`
   зменшується (не реєструє `/analytics`), у новому сервісі з'являється свій
   маленький composition root.

**Що НЕ зміниться:**
- Domain модуля (`UserStats`, `DailyMetric`).
- Application (queries, handlers).
- ACL — той самий клас, лише з парсингом замість прямого доступу.
- `module.js` як точка склейки.

Це показує, чому ціна модульного моноліту окупається: при необхідності
розщеплення на сервіси міняється транспорт і infra, не бізнес-код.

## Частина 2. Ретроспектива курсу 1 → 5

### 2.1. Як змінювалася архітектура на кожному кроці

| Lab | Стан | Ключова зміна |
|---|---|---|
| 1 | `server.js` + `routes/{auth,items}.js` + два global `[]`-масиви | «Big Ball of Mud»: вся логіка в роутах |
| 2 | 4 шари: Presentation/Application/Domain/Infrastructure, ADR на Rich Model | Виділили домен, ввели VOs, factory, repository pattern, доменні помилки |
| 3 | Application розщеплено на `commands/` і `queries/`, з'явилися Read Models | CQS: різні стилі для запису і читання, окремий `IItemReadRepository` |
| 4 | Audit як побічна операція; реалізовано sync і async, in-process Event Bus | Integration Events як публічний контракт між модулями; ідемпотентність |
| 5 | Three bounded contexts (Core/Audit/Analytics), ACL у консумерах | Композиція через композиційний корінь; модулі ізольовані тестом |

Кількісно: lab 1 був ~5 файлів, lab 5 став ~70 файлів. Це не «надмірне
ускладнення» — це **зробила видимим те, що раніше було неявним**: де
домен, де адаптери, де контракти між модулями.

### 2.2. Які архітектурні рішення виявилися найціннішими

**1. Rich Domain Model з VOs (lab 2).**
Найдешевша інвестиція з найвищим ROI. `PasswordHash` як VO з `toJSON()` —
це 20 рядків коду, які гарантовано виключають клас помилок «хеш у логах».
Інваріант власника всередині `Item.rename()` — це 5 рядків, які
гарантовано виключають «забув перевірити» у двох роутах.

**2. Repository pattern + DIP (lab 2).**
Уся доменна логіка тестувалася ~миттєво через in-memory репозиторії.
До п'ятої лаби unit-тести виконувалися без жодної зовнішньої залежності
у середньому за ~400 мс на ~70 тестів. Це масштабована економія часу
на ревью-цикл.

**3. CQS на рівні Application (lab 3).**
Розщеплення commands/queries дало природне розщеплення тестових стилів
(unit для команд, integration для запитів). Окремий read-репозиторій з
`JOIN` для денормалізації виявився чистим способом мати UI-зручні DTO
без шкоди для write-моделі.

**4. Integration Events як публічний контракт (lab 4).**
Це була ментально-зсувна штука. Як тільки Core почав публікувати «факти», а
не «викликати методи», з'явилася можливість додавати модулі без зміни
відправника. Lab 5 — це по суті розкриття можливостей, які lab 4 заклала.

**5. ACL у консумерах (lab 5).**
Стало очевидно, наскільки крихкою буде система, якщо консумери
читатимуть Core-події напряму. Перейменування одного поля Core зламало б
кожного підписника. ACL — це коштує одного класу-перекладача на консумера,
але дає свободу обом сторонам еволюціонувати незалежно.

### 2.3. Що зробив би інакше, знаючи кінцевий результат

**Я б у lab 2 одразу зробив commands/queries.**
У lab 2 я зробив `*UseCase` класи з універсальним `execute()`-методом.
У lab 3 довелося їх масово перейменовувати у `*CommandHandler` /
`*QueryHandler` з `handle()`. Якби я з самого початку знав, що CQS
прийде наступним кроком, я б одразу обрав *Handler-нейминг і
розщеплення write/read директорій.

**Я б додав `EventId` у події з самого початку (lab 4).**
Зараз ідемпотентність працює через ключ `(action, entityType, entityId,
occurredAt)`. Це працює, бо `occurredAt` генерується в Core один раз.
Але якщо подія законно повторюється (наприклад, при retry — той самий
факт може потрапити в чергу двічі), потрібен явний `EventId`. Це
архітектурно правильна штука, яку дешевше додати на самому старті.

**Я б одразу робив **rich domain errors** замість `Error` (lab 2).**
У ранніх версіях lab 2 я кілька разів кидав звичайні `Error` з самописним
повідомленням. Потім переніс на `ValidationError extends DomainError` —
і це дозволило централізовано мапити в 422/409/403. Цей рефакторинг
варто було зробити одразу при першому потребі кидати помилку.

**Я б не вкладав `application/ports/IEventBus` всередину Application
у lab 4.**
Коли в lab 5 знадобилося розкласти на модулі, `IEventBus` довелося
перевозити в `shared/`. З самого початку було б очевидно, що event bus —
це cross-cutting concern, а не специфіка Core Application.

### 2.4. Trade-offs між простотою та гнучкістю

**Lab 1 → lab 5: ціна гнучкості — приблизно 14× більше файлів** (5 → 70).
Час на додавання нової фічі у lab 5 повільніший за lab 1: треба зрозуміти
структуру модуля, написати command + handler + event + ACL у консумерах.

Але **час на безпечну зміну** — навпаки, набагато швидший. Перейменувати
поле в Core у lab 5: 2 файли ACL. У lab 1 те ж саме — кожен файл, де
це поле згадане (з пошуком через grep, з ризиком пропустити).

Концептуально це trade-off **миттєвий час** (lab 1 виграє) проти
**довгострокового часу** (lab 5 виграє). На прототипі, який житиме
місяць — lab 1 правильний. На системі, яка житиме роки і ростиме
кодовою базою — lab 5 правильний.

Найкорисніший інсайт курсу: **архітектура — це не «правильно/неправильно»,
це «що ми оптимізуємо?»** Lab 1 оптимізує час до першого коміту;
lab 5 — час до 100-ї фічі без накопичення техдогу. Обидва відповіді
правильні в своєму контексті, але контекст треба знати свідомо.

### 2.5. Висновок

Курс пройшов шлях від «коду, що працює» до «коду, що тримає форму під
тиском часу і змін». Кожна лаба додавала один структурний елемент —
шари → CQS → події → модулі — і кожен з них вирішував конкретну проблему,
а не був «архітектурою для архітектури».

Найважливіше, що я виніс: **архітектурні патерни — це інструменти для
обмеження дій**. Шар Domain без зовнішніх імпортів обмежує мене від
змішування бізнес-логіки і SQL. ACL обмежує мене від «протікання»
зовнішніх типів. Це обмеження, які я наклав на себе свідомо — і саме
вони роблять код керованим. Без них код був би «свободніший», але
неспокійний для всіх, хто доторкнеться до нього після мене.

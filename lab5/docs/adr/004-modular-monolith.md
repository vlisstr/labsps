# ADR-004: Розщеплення на 3 модулі — Core, Audit, Analytics

- Статус: Прийнято
- Дата: 2026-05
- Контекст: Лабораторна робота №5 — модульний моноліт з bounded contexts.

## Контекст

До lab 5 проєкт був одним великим контекстом: усі команди, події, репозиторії
жили в одному `src/application/` і `src/domain/`. Це працювало, бо домен
маленький (User + Item), але:

- audit з lab 4 жив поряд з командами Core і ніщо не заважало комусь у
  майбутньому імпортувати, наприклад, `Item` у subscriber audit;
- analytics-фунціонал, якби з'явився, теж сів би «у спільну купу»;
- межі модулів існували лише в уяві програміста — компілятор не допомагав.

## Рішення

Виділити **три bounded contexts**, кожен у власній папці `src/modules/X/`,
з повною внутрішньою архітектурою (domain, application, infrastructure)
і єдиним публічним контрактом — `module.js`:

### 1. Core
**Відповідальність**: бізнес — реєстрація користувачів, авторизація, CRUD на
items з інваріантами власника.

**Публічний контракт**: HTTP routes (`/auth/*`, `/items/*`) + публікація
Integration Events (`UserRegistered`, `ItemCreated`, `ItemUpdated`,
`ItemDeleted`).

**Стан**: master-of-truth для User та Item.

### 2. Audit
**Відповідальність**: append-only журнал «хто, коли, що зробив».
Read-only consumer Core-подій.

**Публічний контракт**: HTTP route `GET /audit`. Назовні нічого не публікує.

**Стан**: власна модель `AuditEntry` з власною ідемпотентністю.

### 3. Analytics
**Відповідальність**: агрегована статистика — топ користувачів за items,
щоденна активність, загальні лічильники. Read-only consumer.

**Публічний контракт**: HTTP routes `GET /analytics/{summary,users,daily}`.

**Стан**: ВЛАСНІ агрегати — `UserStats`, `DailyMetric` — а не копії
доменних `User`/`Item`. Це принциповий момент модульного моноліту.

## Альтернативи, які розглядалися

### A) Залишити моноліт одним контекстом (як lab 4)
- Простіше, менше boilerplate.
- Але не задовольняє ціль лаби й не дає захисту від «розпливання» меж.

### B) Два модулі (Core + Analytics), audit — частина Core
- Лаба дозволяє 2–3.
- Але audit структурно ідентичний Analytics: ACL з Core-подій у власну
  модель, read-only. Об'єднання з Core створило б асиметрію.

### C) Виділити Auth в окремий модуль
- Логічно (auth — окрема концепція в DDD), але технічно auth тісно
  пов'язана з User aggregate. Розщеплення створило б проблему spanning
  aggregate.

### D) Микросервіси (виділити процеси)
- Зовсім інша задача. У ції лабі робимо модульний моноліт як
  «передостанній крок до мікросервісів».

Обрано **3 модулі**: Core, Audit, Analytics — як C.

## ACL (Anti-Corruption Layer)

Кожен консумер має власну ACL, яка ТРАНСЛЮЄ Core-події у внутрішні
типи:

- **Audit/ACL** — `auditEventHandlers.js`. `ItemCreated` → `new AuditEntry({
  action: 'item.created', entityId: event.itemId, details: { name: event.name } })`.
- **Analytics/ACL** — `CoreEventTranslator.js`. `ItemCreated` → виклик
  `userStats.recordItemCreated(occurredAt)` + `dailyMetric.incItemsCreated()`.

Структура події Core (`type`, `itemId`, `name`, `ownerId`) не «протікає»
далі ACL. У внутрішніх типах Analytics і Audit немає поля з назвою
`itemId` — є `entityId` (audit) або інкремент лічильника (analytics).

Тому якщо завтра Core перейменує `itemId` → `id` у події, зламається
лише ACL — два файли, добре локалізована зміна.

## Strong vs Eventual Consistency

| Місце | Тип | Чому |
|---|---|---|
| У межах одного агрегату Core (`Item.rename` → `repo.save`) | **Strong** | Одна транзакція, інваріанти захищають домен |
| Core → Audit | **Eventual** | Подія асинхронна, audit оновлюється після Core, час відставання — мілісекунди в моноліті |
| Core → Analytics | **Eventual** | Те ж саме; для лічильників затримка некритична |
| GET /analytics/summary | відповідає миттєво на найсвіжіший знаний стан Analytics | Може показати трохи старі дані, якщо подія ще «в дорозі». Це by design і прийнятно |

Чому це **прийнятно для бізнесу**:
1. Аудит — пост-фактум по визначенню; читач (security/support) не
   запитує його в одній мс після операції.
2. Analytics — звітність. Користувач дивиться дашборд, не чекає
   реальний час. Затримка в секунди некритична навіть для тих сценаріїв,
   де аналітика динамічна.
3. Core — джерело правди для бізнес-операцій. Він не залежить від
   доступності Audit чи Analytics: якщо їхні підписники впадуть,
   Core продовжує приймати запити (verified тестом).

## Наслідки

- З'являється `composition_root.js` як єдиний знавець про всі модулі.
- `module.js` кожного модуля експортує тільки `buildXModule({ eventBus })`
  → `{ routes, ... }`. Решта файлів модуля внутрішні.
- Інтеграційні тести між модулями використовують `eventBus` напряму, а не
  HTTP — це швидко і показує справжню точку контракту.
- Перевести модуль у окремий сервіс легко: `InProcessEventBus` замінюється
  на `RabbitMQEventBus` (інтерфейс той самий), routes Audit/Analytics
  виносяться в окрему точку входу. Жоден файл всередині модуля не
  змінюється.

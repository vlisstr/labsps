# ADR-002: CQS на рівні Application Layer

- Статус: Прийнято
- Дата: 2026-05
- Контекст: Лабораторна робота №3 — розщеплення запису і читання.

## Контекст

У lab 2 Application Layer складався з «use case»-ів: один клас на одну
бізнес-операцію (`CreateItemUseCase`, `ListItemsUseCase`, …). Це вже краще
за «один великий ItemService», але між операціями запису та читання
залишалося приховане змішування:

- `ListItemsUseCase` повертав `Item[]` через `itemToView()`-маппер у застосунковому шарі — це
  доменна модель, перетворена в DTO «на льоту».
- Той самий `IItemRepository` обслуговував і команду (`save`, `findById`),
  і запит (`findAll`).
- У контролерах `POST /items` повертав повну модель з усіма полями (зокрема
  `ownerId`), хоча клієнту після створення зазвичай достатньо лише `id`.

Це створює дві проблеми, описані у CQS / Repository-патерні:

1. **Сервіс із подвійною відповідальністю.** Якщо завтра з'явиться вимога
   "GET /items має повертати ще й ім'я власника та статистику переглядів",
   то це потрапить у `IItemRepository.findAll` і він почне рости в бік UI.
2. **Тести змішують режими.** Тестуючи `ListItemsUseCase`, ми все одно
   ходимо через мапер доменної моделі, хоча для запиту нам не потрібен жодний інваріант.

## Розглянуті варіанти

### A) Залишити use cases як у lab 2
- Зрозуміло, мало коду.
- Але read-сторона завжди обмежена формою доменної моделі.

### B) Повний CQRS (різні БД / репліки / event projections)
- Надмірно для двох сутностей і простого CRUD.
- Eventual consistency додає купу складності, якої тут не треба.

### C) **CQS на одній БД (обрано)**
- Розділити Application Layer на `commands/` і `queries/`.
- Один `CommandHandler` = одна операція запису. Повертає лише `id` (або нічого).
- Один `QueryHandler` = одна операція читання. Повертає `*ReadModel` (DTO),
  оптимізований під споживача.
- Окремий `IItemReadRepository` (інтерфейс у `application/queries/`), реалізації
  в інфраструктурі: in-memory join і SQL JOIN.

## Рішення

Обрано **C: CQS на рівні Application**. Це дешева зміна (одна-дві години
рефакторингу), що дає більшість переваг повного CQRS:

- Контролери стають «маппінговими»: HTTP → Command/Query → handler;
- Тести команд і запитів отримують різні підходи: команди — unit з fake-repo,
  запити — unit (через fake read-repo) + integration (реальний SQL).
- Read-сторона може еволюціонувати незалежно: `ownerUsername` додано через
  `LEFT JOIN users` у `SqliteItemReadRepository` — write-сторона про це не знає.

## Класифікація `Login` як команди

`POST /auth/login` не змінює персистентний стан, але:
- Видає токен (артефакт виконання, як ID для `Create`).
- Логічно є «дією», яку клієнт хоче виконати, а не «вибіркою даних».
- У майбутньому може отримати побічні ефекти: лічильники невдалих спроб,
  блокування акаунта, audit log.

Тому `LoginUserCommand` залишається командою, а не запитом. Це
прагматичний відступ від буквального прочитання CQS, який CQS-документ
Greg Young'а явно дозволяє.

## Наслідки

- В `application/` з'явилися дві окремі підпапки `commands/` і `queries/`.
- Доменний `IItemRepository` втратив метод `findAll()` — він не потрібен домену.
- Read-моделі (`ItemListItemDTO`, `ItemDetailDTO`) живуть у `application/dto/`
  і повертаються запитами як готові структури з денормалізованими полями.
- Контролери більше не маплять доменні об'єкти на JSON вручну —
  команди повертають `id`, запити повертають DTO, і обоє йдуть прямо у відповідь.
